# Plan

## Current goal

- ex01~ex03 REPL 예제의 실행 경로를 유지하면서 회귀를 줄일 최소 검증 체계를 추가한다.
- 팝업/새 창 추적과 selector 대기 정책 같은 취약 지점을 코드와 문서에서 더 명확히 다듬는다.

## Near-term work

- `browser.py`의 명령 파서와 `wait`/`save_yaml`/`download_links`를 분리 테스트할 수 있는 단위 테스트 뼈대를 만든다.
- 새 창 감지를 현재 폴링 방식에서 Playwright의 popup/page 기대 API로 바꿀지 검토한다.
- 루트 `main.py`가 ex03까지 다룰지, 아니면 ex01/ex02 래퍼로만 둘지 엔트리포인트 전략을 명확히 결정한다.
- `.playwright/system.yaml` 샘플을 저장소에 안전하게 유지할지, 별도 예제 파일로 분리할지 정리한다.

## Structure decisions

- 애플리케이션 코드는 `src/playwrite_tutorial/` 아래에 둔다.
- 브라우저 실행/종료 책임과 REPL 입력 처리 책임을 분리한다.
- 초기에는 "단일 브라우저 세션 + 순차 명령 실행" 모델로 단순하게 시작한다.
- 현재 기본 엔트리포인트는 `playwrite_tutorial.ex01:main`이며, 루트 `main.py`는 보조 실행 래퍼로만 유지한다.
- 예제 스크립트나 샘플 명령은 재사용 가능하도록 별도 모듈 또는 `examples` 성격의 파일로 분리한다.
- 루트의 `main.py`는 가능하면 얇은 실행 진입점으로만 유지하고 핵심 로직은 패키지 내부로 옮긴다.
- 로컬 설정 파일과 실행 산출물은 기본적으로 `.playwright/` 아래에 둔다.

## Risks

- Playwright Python 패키지 설치와 Firefox 브라우저 바이너리 설치는 별도 단계라서 초기에 실행 오류가 날 수 있다.
- Firefox 전용 동작은 사이트별 로딩 타이밍과 셀렉터 안정성에 영향을 많이 받는다.
- 자연어에 가까운 명령 인터페이스로 바로 넓히면 범위가 커질 수 있으므로 초반에는 고정된 명령 형식이 필요하다.
- GUI 환경 여부에 따라 headed 실행이 제한될 수 있으니 headless 옵션도 함께 고려해야 한다.
- 새 창이나 iframe 로더를 거치는 페이지는 도달 경로가 불안정할 수 있다.
- 캔버스 렌더링 UI는 DOM 기반 추출이 거의 되지 않아 시각 인식 품질에 의존하게 된다.
- 현재 새 창 감지 루프는 최대 1.5초만 폴링하므로 늦게 뜨는 팝업은 놓칠 수 있다.
- 자동화 테스트 부재 때문에 리팩터링 시 REPL 동작이 조용히 깨질 수 있다.
